## プロダクト概要

### 名称
**漢字合体ガチャ（仮称）** - ガチャ × 漢字合体のスナック型Webゲーム

### プロダクトコンセプト
- **ガチャ × 漢字合体**: ガチャで引いた部首・つくりを、プレイヤー自身の「ひらめき」で組み合わせて漢字を作り出す。運要素と発見の快感を両立する。
- **スナック型スコアアタック**: 1プレイ約1〜2分。ガチャ残り回数制の耐久型で、いつでも手軽に・短時間で完結する。
- **クライアント完結・低コスト**: 漢字判定はすべてアプリ内のJSON辞書で完結。サーバー0台で動作し、運用コスト・セキュリティリスクを最小化する。

### プロダクトビジョン
誰でも「漢字を組み立てる楽しさ」と「神引き・事故引きの一喜一憂」を、説明なしに数十秒で体験できるカジュアルゲームを提供する。
プレイヤーは部品をガチャで引き、自分のひらめきで漢字を完成させ、その瞬間の気持ちよさと「へえ、こうやってできるんだ」という発見をSNSで共有したくなる。
まずWebで「面白さの証明」を行い、人気が出たらネイティブ／収益化へ段階的に拡張する。サーバーレスを貫くことで、個人開発者が赤字リスクゼロで永続的に運営できる。

### 目的
- カジュアル層が即理解・即プレイできる、短時間完結のゲーム体験を提供する
- 「漢字を完成させる快感」と「発見（学び）」を、ストレスなく届ける
- 運用・セキュリティのコストとリスクを最小に保ち、個人開発で持続可能にする
- Web版で人気を検証し、ネイティブ・収益化へ無理なく接続できる土台を作る

## 成功指標(KPI)

### プライマリーKPI（プレイ体験／リリース後3ヶ月）
- **5プレイ目までの離脱率**: 30%以下（コアループが刺さっているかの最重要指標）
- **即時リトライ率**: 40%以上（結果画面から「もう一回」を押す割合）
- **1ゲームの平均合体回数**: 5〜8回（手軽さと手応えのバランス指標）
- **SNSシェア発生率**: 1セッションあたり5%以上（バズの先行指標）

### セカンダリーKPI
- **「詰み終了」率**: 10〜20%（高すぎ＝理不尽、低すぎ＝緊張感不足。プール設計の健全性指標）
- **ヒント利用率**: 30〜50%（知識の壁を親切UIで吸収できているか）
- **リテンション率**: 翌日20%以上 / 7日10%以上（3ヶ月後）
- **DAU**: リリース後3ヶ月で日次500人（Web、オーガニック想定）

### 測定方法

**Web MVP期間中に計測する指標（ローカルログのみ・送信なし）**
- 1プレイごとに記録：合体回数・ミス回数・ヒント利用回数・交換利用回数（旧・捨てる。内部フィールド名は互換のため `discardUsed` を維持）・終了理由（詰み／手札0）・最終スコア・新発見漢字数・プレイ所要時間
- これらから「平均合体回数」「詰み終了率」「ヒント利用率」「即時リトライ率」「5プレイ離脱率」をクライアント内で集計する
- 「詰み終了率」の定義：F6終了条件の (a)＋(b) のANDで終了したケースを「詰み」とカウントする（手札0＋ガチャ残0は別カウントとする）

**ネイティブ移行時に追加する指標**
- DAU・リテンション・SNSシェア発生率など、集計・送信が必要な指標は、プライバシーに配慮した最小限の匿名計測をネイティブ移行時に導入する（Web MVPでは任意）

**ネイティブ移行・収益化トリガー（暫定値・要調整）**
- 整理書5.5「トリガーを先に数字で決める」原則に基づく暫定基準：
  **「DAU 500人超 かつ 7日リテンション15%超 を 30日継続」したら、ネイティブ化・本格収益化の検討を開始する**

## 機能要件

### コア機能(MVP)

#### F1. ガチャ機能

**ユーザーストーリー**:
プレイヤーとして、運の一喜一憂を楽しむために、ボタン一つで部品を引けるガチャが欲しい。

**受け入れ条件**:
- [ ] 「ガチャを回す」操作で、現在のレベルのプールから部品が1つ抽選され、手札に追加される
- [ ] 部品は★1〜★5のレアリティを持ち、レベル別の出現率テーブルに従って抽選される（出現率テーブルとガチャプール構成の具体値は機能設計書 functional-design.md で定義する。整理書3.5に初期案あり）
- [ ] カプセル降下→開封→部品出現の一連アニメーションが再生され、レアリティ別の色・バースト演出が出る
- [ ] ガチャ残り回数が画面に常時表示され、1回引くごとに減る
- [ ] ゲーム開始時、手札は上限（または引ける残数）まで**自動で補充**される（初期手札。手動で1枚ずつ引き始める必要はない）
- [ ] 手札が上限（暫定12、要調整）に達している場合は、引く前に手札整理を促す

**優先度**: P0(必須)

#### F2. 合体機能

**ユーザーストーリー**:
プレイヤーとして、ひらめいた漢字を完成させる快感のために、手札の部品を選んで合体できる仕組みが欲しい。

**受け入れ条件**:
- [ ] 手札から部品を2つ以上選択し、「合体！」操作で判定が走る
- [ ] 判定はアプリ内JSON辞書の検索で行い、有効な漢字なら成功、なければミスとなる
- [ ] 部品の配置（左右・上下）は不問で判定する
- [ ] 多部品の合体を許容する（例：木+壴+寸=樹）。部品数に上限を設けない（手札の範囲内）
- [ ] 複数解を持つ組み合わせは、いずれも正解として扱う（例：日+月=明/朙）
- [ ] 成功時、使用した部品が手札から消費される
- [ ] 成功演出（白フラッシュ・漢字の大表示・リング/パーティクル・読み/意味表示・スコアフロート）が再生される
- [ ] 失敗時はコンボがリセットされ、赤シェイク・✕表示が出る（部品は消費しない）

**設計メモ**:
- 「多部品OK」は許容であって強制ではない。カジュアル勢は2部品で、上級者は多部品で自然に差がつく設計。多部品選択を制限する仕様は入れない。
- 「作った漢字をさらに素材にして合体する」連鎖は本MVPでは実装しない（将来モード：F16）。本機能の合体対象は手札の部品のみとする。

**優先度**: P0(必須)

#### F3. レベル選択

**ユーザーストーリー**:
プレイヤーとして、自分の実力や気分に合わせるために、難易度レベルを選びたい。

**設計原則（本ゲームの中核思想・必読）**:
難易度は「①有効辞書の範囲 ②ガチャプール構成 ③ヒント開放度 ④出現率の偏り」の組み合わせで制御する。合体に使える部品数を制限したり、多くの救済機能で後追い補修することはしない。特にやさしいモードでは、手札に `木+木=林`・`日+月=明` 級の2部品正解が構造的にほぼ常に1つ含まれるようプールを構成し、詰みをそもそも発生させない（床の保証）。この思想はF5（救済を2つに絞れる根拠）と表裏一体である。**ガチャプールは単なるレアリティテーブルではなく、難易度設計の主役として実装すること。**

**受け入れ条件**:
- [ ] ホーム画面で「やさしい／ふつう／むずかしい」を選択できる
- [ ] レベルにより、有効辞書の範囲が切り替わる（やさ=教育漢字約1,026字／ふつう=中学卒業相当約1,600字／むず=常用漢字2,136字）
- [ ] レベルにより、ガチャプール構成・出現率・ヒント開放度・交換コストが切り替わる
- [ ] 「やさしい」では、手札に2部品の有名漢字の正解が高確率で含まれるようプールが構成される（床の保証）
- [ ] 難読漢字（常用外）は出題されない（極モードは本MVPのスコープ外）

**優先度**: P0(必須)

#### F4. スコア・コンボ

**ユーザーストーリー**:
プレイヤーとして、上達と競争を楽しむために、合体の価値が点数に反映されてほしい。

**受け入れ条件**:
- [ ] 合体成功時、作った漢字の画数 × 基礎係数1 をそのまま基礎点とする（例：林8画→8点）※係数は要調整
- [ ] 基礎点にコンボ倍率を乗算してスコアに加算する（加点 = 基礎点 × コンボ倍率）
- [ ] コンボ倍率は連続成功で上昇する（暫定：1.0→1.5→2.0→3.0、上限3.0倍。段階の刻みは要調整）
- [ ] ミス（無効な合体）でコンボ倍率が1.0にリセットされる
- [ ] 現在スコアとコンボ状態（現在の倍率）が画面に常時表示される
- [ ] 多部品・高画数の合体ほど高得点になる（画数ベースにより自己バランスする）

**優先度**: P0(必須)

#### F5. 救済（ヒント／交換）

**ユーザーストーリー**:
プレイヤーとして、詰まってもストレスなく続けるために、最小限の打開手段が欲しい。

**受け入れ条件**:
- [ ] 「ヒント」操作で、合体できるペアが1組やさしく点滅表示される
- [ ] ヒントの利用可否・コストはレベルに従う（コスト単位はガチャ残り回数）：やさ=無料・常時／ふつう=ガチャ残-1（暫定、要調整）／むず=利用不可
- [ ] 「交換」操作（旧・捨てて引き直す）で、選択した手札 **1枚以上** を一括で引き直せる（交換後の手札枚数は不変）
- [ ] 達成型（deck）では交換した部品を山札へ戻して切り直す（部品は場から失われず、全対象字を完成できる状態が保たれる）
- [ ] 交換のコストはレベルに従い、**1枚あたり×枚数**を消費する（コスト単位はガチャ残り回数）：やさ=無料（消費なし）／ふつう=ガチャ残-1／むず=ガチャ残-2。総コストが残不足の場合は実行しない（消費もしない）
- [ ] v1.0のピティ天井・相性ボーナス・レスキューガチャ・再振りチケットは実装しない（プール設計で詰みを予防する。F3設計原則と表裏一体）

レベル別コスト一覧（コスト単位＝ガチャ残り回数。暫定値・要調整）：

|  | やさしい | ふつう | むずかしい |
|---|---|---|---|
| ヒント | 無料・常時 | ガチャ残-1 | 利用不可 |
| 交換（1枚あたり） | 無料 | ガチャ残-1 | ガチャ残-2 |

**優先度**: P0(必須)

#### F6. 耐久セッション・結果画面

**ユーザーストーリー**:
プレイヤーとして、短時間で区切りよく遊ぶために、ガチャ残り回数制の1プレイと結果表示が欲しい。

**受け入れ条件**:
- [ ] 1プレイはガチャ固定10回（暫定値、要調整）から開始する
- [ ] 以下の両方が揃ったタイミングでゲーム終了となる：(a) ガチャ残り回数が0、かつ (b) 現在の手札に合体可能な組み合わせが1つも存在しない（手札の全ペア／多部品の組み合わせを辞書総当たりして有効解なし）
- [ ] 手札0枚かつガチャ残0の場合も終了に含む
- [ ] ガチャ残0でも手札に合体可能な組み合わせが残る場合はゲームを継続する
- [ ] 終了判定は、各「ガチャ」「合体」「交換」操作の完了後に実施する
- [ ] 結果画面に、最終スコア・作成した漢字一覧・新発見の漢字・称号・「もう一回／シェア／ホーム」を表示する
- [ ] 称号は、バックエンドを使わずローカルの固定スコア閾値テーブルで算出する（実際のプレイヤー分布は参照しない）。表示は**ランク制**とし、リアルタイム順位を意味する「上位○%」表記は誤解を招くため用いない（例：スコア帯ごとに 見習い／中級／上級／名人。閾値・ランク名は要調整）
- [ ] 1プレイの体感時間が概ね1〜2分に収まる

**優先度**: P0(必須)

#### F7. 図鑑

**ユーザーストーリー**:
プレイヤーとして、継続して遊ぶ理由のために、作った漢字が記録・収集されてほしい。

**受け入れ条件**:
- [ ] 合体に成功した漢字が図鑑に記録される
- [ ] 各漢字について、文字・読み・意味（KANJIDIC2由来）を表示する
- [ ] 収集率を総数表示する（例：850 / 2,136）
- [ ] 図鑑データはローカルストレージに永続保存され、再訪時に保持される

**優先度**: P0(必須)

#### F8. シード式デイリー（今日のお題）

**ユーザーストーリー**:
プレイヤーとして、毎日戻る理由のために、その日限定の共通課題で遊びたい。

**受け入れ条件**:
- [ ] 日付をシード値とした疑似乱数で、その日の部品セット／ガチャ結果が決定される（サーバー不要）
- [ ] 同じ日付なら全プレイヤーが同一条件（同じレベル・同じガチャ列）になる
- [ ] デイリーのレベルは日替わりで固定とし、その日の対象レベルを画面に明示する（暫定仕様、要調整。フリープレイのレベル選択とは独立）
- [ ] 1日1回を基本とし、再挑戦の可否を定義する（暫定：その日のベスト更新を狙って再挑戦可）
- [ ] 「今日のお題」のベストスコアをローカル保存し、更新を表示する
- [ ] グローバルランキング（他者とのスコア比較）はスコープ外（後期・要BE）

**優先度**: P0(必須)

#### F9. ベストスコア記録

**ユーザーストーリー**:
プレイヤーとして、自分の上達を実感するために、過去最高スコアが残ってほしい。

**受け入れ条件**:
- [ ] レベル別のベストスコアをローカル保存し、ホーム画面で確認できる
- [ ] 新記録達成時に結果画面で明示する

**優先度**: P0(必須)

#### F10. ライセンスクレジット表記

**ユーザーストーリー**:
運営者として、ライセンス遵守のために、データソースのクレジットを必ず表示したい。

**受け入れ条件**:
- [ ] 設定／About画面に、KANJIDIC2・KRADFILEのクレジットとCC BY-SA 4.0表記を掲載する
- [ ] 表記内容は整理書6.1の雛形に準拠する

**優先度**: P0(必須)

#### F11. 結果シェア

**ユーザーストーリー**:
プレイヤーとして、神引き・好成績を自慢するために、結果を簡単に共有したい。

**受け入れ条件**:
- [ ] 結果画面から、スコアと作成漢字を含むシェア用テキストを共有できる
- [ ] シェアテキストは「神引き・事故引きでネタになる」よう、レベル・スコア・作成漢字・ハッシュタグを含む（書式例：`漢字合体ガチャ｜むずかしい｜スコア142｜作った字：林・明・好・樹 ほか3字 #漢字合体ガチャ`）※書式は要調整
- [ ] Web Share API を基本とし、非対応ブラウザではテキストのクリップボードコピーにフォールバックする
- [ ] 追加サーバーなしで実現できる方式を用いる（結果画像の生成はP1の将来機能）

**優先度**: P1(重要)

## 非機能要件

### パフォーマンス
- 合体判定（辞書検索）: 1ミリ秒以下
- 初回起動時の辞書ロード: 2秒以内（メモリ展開して以降は高速判定）
- 内蔵辞書データ: 常用漢字2,136字対応でJSON圧縮後 約100KB以内

### ユーザビリティ
- 新規プレイヤーが、説明を読まずに最初のプレイを開始できる（ホームから2タップ以内でゲーム開始）
- スマホ・PC両対応のレスポンシブUI
- ライト／ダークの視認性に配慮（最低限、暗所でも視認可能なコントラスト）

### 動作環境
- 対象ブラウザ：モダンブラウザの最新版（Chrome / Safari / Firefox / Edge）
- 対象デバイス：スマートフォン（縦持ち）を主、PCを副とする
- F11のWeb Share APIはブラウザ差異があるため、非対応時はクリップボードコピーにフォールバックする（F11参照）

### 信頼性
- 完全オフライン動作（ネットワーク切断時もガチャ・合体・記録が正常に動く）
- 図鑑・ベストスコア等のローカルデータが、リロード・再訪後も失われない

### セキュリティ／プライバシー
- クライアント完結。ゲームプレイに関する外部サーバー送信を行わない
- MVPでは広告SDKを導入しない（プライバシー・セキュリティ面を増やさない）
- 個人情報・アカウント登録を要求しない

### ライセンス遵守
- データのCC BY-SA 4.0継承条件を満たす（クレジット表記、改変データの同ライセンス公開）
- 自作合体辞書もCC BY-SAで公開する（安全策）

### スケーラビリティ／コスト
- Web版はサーバー0台・運用コスト0円で成立する
- 将来のオンライン機能は無料枠（Firebase等）で数千ユーザーに対応できる構成を前提とする

---

## スコープ外

本MVP（Web版第1リリース）では、以下を明示的にスコープ外とする：

- **グローバルランキング**（バックエンドが必要。F14として後期）
- **課金ガチャ／有料ランダム要素**（規制・炎上リスクのため永久に不採用）
- **常用外の難読漢字**（極モードまで保留。F15）
- **連鎖・お題・二字熟語・対戦などの追加モード**（F16）
- **ネイティブアプリ／PWAのオフラインインストール**（F18として後期。MVPはブラウザ動作）
- **日本語以外の多言語対応**
- **アカウント／クラウド同期**（ローカル保存のみ）
- **収益化機能**（Web版は投げ銭リンクのみ検討、本格収益化はネイティブ後）
