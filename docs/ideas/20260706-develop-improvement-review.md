# develop ブランチ実装レビュー・改善提案（2026-07-06）

develop ブランチ（`149c696` 時点）のソースコード・設定・ドキュメントを、
色合い / 演出 / UI・UX / アクセシビリティ / セキュリティ / 免責・法務 / PWA・パフォーマンス / 運用
の観点でレビューした結果と改善提案。優先度は ★★★（早めに対応推奨）〜★（余裕があれば）。

> 注: さくら背景等のビジュアル刷新・ボタン UX は未マージの
> `feature/20260623-button-ux-system`、交換の一括化・初期手札自動補充は PR #86 にあり、
> 本レビューは develop 時点の実装を対象とする。

## 0. 総評（良い点）

先に、現状よく出来ている点を記録しておく（改善時に壊さないこと）。

- **CSP が厳格**（`public/_headers` を単一の正とし、`default-src 'self'`・`frame-ancestors 'none'`・
  HSTS・`X-Content-Type-Options` 等を網羅）。外部 CDN・外部 API 依存ゼロでローカル完結。
- **XSS 面が堅い**: `{@html}` 不使用（Svelte の自動エスケープに一貫)、外部リンクは
  `rel="noopener noreferrer"` 徹底、共有はテキストのみ。
- **localStorage の防御的読込**: `migrate()` が信頼しない入力を正規化し、破損時は既定値へ
  フォールバック。プライバシーモードでも `StorageUnavailableError` → メモリ動作でクラッシュしない。
- **`prefers-reduced-motion` 対応**が effects.css / ParticleField / 装飾に一貫。
- **データ出典クレジット**（KANJIDIC2 / KanjiVG / CC BY-SA 継承 / Yuji Syuku OFL）を About に常設。
- ふりがな・文字サイズ拡大・読み上げ（TTS・ローカル完結）と、学習配慮の設定が揃っている。
- 決定性設計（シードデイリー・RNG 注入）とテスト（ユニット 260・E2E 41・カバレッジ閾値）が良好。

## 1. 色合い（カラー・テーマ）

### 1-1. 金色テキストのコントラスト不足 ★★★
`--md-sys-color-tertiary: #b8860b`（金）は和紙背景 `#f5f1e6` に対し **約 2.9:1** で、
WCAG AA（通常文字 4.5:1）に不適合。大文字（3:1）でもぎりぎり届かない。
ベストスコアの金数字・高レアリティ表示など「テキストとして」使う箇所が対象。

- 対応案: テキスト用の金を `#8a6508` 程度まで暗くしたトークン
  （例 `--md-sys-color-tertiary-text`）を追加し、`#b8860b`/`#d4af37` は装飾（罫線・グロウ・印）に限定する。
  もしくは金テキストに `text-shadow`/縁取りでコントラストを補強する。

### 1-2. ダークモード未対応 ★★
`prefers-color-scheme` の分岐が無い。トークンは `material-tokens.css` に集約済みなので、
`@media (prefers-color-scheme: dark)` で夜間パレット（墨背景×生成り文字×抑えた朱）を
追加するコストは小さい。子どもの就寝前利用・OLED 端末を考えると価値が高い。
`theme-color` メタも `media` 属性で 2 色併記する。

### 1-3. 強制カラーモード（Windows ハイコントラスト）★
`forced-colors: active` で `border`/`outline` が生き残るかは未検証。かるた札（PartChip）の
選択状態が背景色だけに依存しないこと（現状 border+aria-pressed ありなので概ね可）を確認し、
`forced-color-adjust` の明示が必要な箇所だけ手当てする。

### 1-4. レアリティの色分けと色覚多様性 ★
藍（★1-2）/朱（★3）/金（★4-5）の色分けは、落款ラベル（「珍」等）併記により
色だけに依存しない設計になっており概ね良い。グロウ強度のみの差は識別性が弱いので、
ラベルの常時表示を維持すること（消さない）。

## 2. 演出

### 2-1. 演出量の設定（スキップ/短縮）が無い ★★
巻物リビール（EmakimonoReveal）・筆順アニメは初見では効果的だが、リピーターや
大人モード・タイムアタックでは冗長になりうる。`prefers-reduced-motion` とは別に、
アプリ内設定「えんしゅつ: ふつう/すくなめ」を追加し、リビール時間短縮・
パーティクル削減を選べるようにする（設定は `settings` に追加、`migrate` 補完で後方互換）。

### 2-2. 効果音・触覚フィードバックが無い ★★
現状の音は TTS 読み上げのみ。合体成功/ミス/ガチャの短い効果音（自前ホスト・
WebAudio 合成なら追加アセット不要で CSP も無傷）と、モバイルでの
`navigator.vibrate`（ミス時 1 パルス等）は学習ゲームの手応えを大きく高める。
既定 OFF か、既存の TTS トグルと並ぶ「おと」トグルに統合し、必ず設定で切れるようにする。

### 2-3. タイムアタック終盤の緊迫演出 ★
残 10 秒/5 秒の警告は TimeBar の色変化のみ。音（設定連動）・画面周縁の
ビネット点滅など「見なくても分かる」チャンネルを 1 つ足すと時間切れの
納得感が上がる。`reduced-motion`/音設定を尊重すること。

### 2-4. 良い実装の維持 （参考）
- パーティクルは固定プール再利用で 60fps 死守の設計 — 維持。
- ミス時の kg-shake は 450ms・過度でない — 維持。
- 演出は判定をブロックしない（Canvas 独立 rAF・リビールは装飾オーバーレイ）— 維持。

## 3. UI・UX

### 3-1. フィードバック行に aria-live が無い ★★★
`feedback`（「+12！」「もう作ったよ」「✕ ミス…」等）はゲームの主要な結果通知だが、
プレーンな要素への文字列代入のみで、スクリーンリーダーに通知されない。
`role="status"`（暗黙 `aria-live="polite"`）を付与する。手札満杯の `.organize` は
`role="status"` 済みなので、これと揃える。

### 3-2. フォーカス可視スタイルが皆無 ★★★
`src/` 全体に `:focus`/`:focus-visible` の指定が **0 件**。ブラウザ既定のリングが
リセットで消えている場合、キーボード操作者は現在位置を見失う。全ボタン共通で
`:focus-visible { outline: 3px solid var(--md-sys-color-secondary); outline-offset: 2px; }`
程度をグローバルに追加する（タブレット＋外付けキーボードの教育現場を想定）。

### 3-3. 誤操作からの回復手段（Undo）が無い ★★
「捨てて引き直す」（PR #86 以降は一括「交換」）は取り消せない。特に一括交換の導入後は
誤タップの影響が拡大するため、スナックバー型の「もとにもどす」（5 秒間・1 回）か、
2 枚以上の交換時のみ軽い確認を挟むことを検討する。ドメインは決定的 RNG のため、
Undo は「交換前の hand/deck スナップショット復元」で実装可能。

### 3-4. チュートリアルの再表示導線が無い ★★
`TutorialOverlay` は初回のみで、`tutorialDone` を戻す UI が無い。Home か About に
「あそびかたをもう一度みる」ボタンを追加する（`setTutorialDone(false)` で再表示するだけ）。
子ども向けアプリでは「後からもう一度見たい」が高頻度で発生する。

### 3-5. 手札の整列・視認補助 ★
手札 12 枚時に同じ部品が離れて並ぶと合体候補を見つけにくい。
「ならべかえ（同じ字でまとめる/レアリティ順）」トグル、または同一部品の枚数バッジを検討。
並べ替えは `instanceId` キーの each を維持すれば演出再生なしで安全に入れられる。

### 3-6. 「合体！」と交換系の同時活性 ★
2 枚以上選択時、「合体！」と交換系ボタンが同時に有効になる（PR #86 の申し送りと同じ論点）。
ボタンの距離・色差で誤操作は少ないと見るが、一括交換導入後に実利用を観察し、
必要なら交換ボタンを合体ボタンから視覚的に一段離す。

## 4. アクセシビリティ（UI と別枠の構造面）

### 4-1. チュートリアルダイアログにフォーカストラップが無い ★★
`role="dialog"` `aria-modal="true"` は付与済みだが、Tab が背景コンテンツへ抜ける。
表示中は overlay 内で Tab を巡回させ、Esc で閉じる（スキップ扱い）を追加する。
背景に `inert` を付けるのが最小実装（Svelte なら `{#if}` と併用で容易）。

### 4-2. 見出し階層・ランドマークの点検 ★
`h1`（アプリ名）→ 各画面 `h2` は概ね良い。`nav`/`main` ランドマークは使えているが、
スキップリンク（「本文へ」）が無い。単画面 SPA なので優先度低。

### 4-3. TTS の発話率・音量 ★
`speak()` は rate/pitch 未指定（既定 1.0）。低学年向けにやや遅め（rate 0.9）を既定にするか、
設定で「よみあげのはやさ」を持つと学習用途に合う。

## 5. セキュリティ

### 5-1. URL レバーが本番でも有効 ★★
`?seed=` / `?taMs=` / `?deckMax=` が本番ビルドでも効く。現状の被害は
**自分の localStorage のベストスコア偽装のみ**（サーバー送信なし）で許容範囲だが、
以下を推奨:
- `?taMs`/`?deckMax` 使用時はセッションを「記録対象外」とし、ベスト更新・図鑑反映を
  スキップする（`?seed` はデイリー再現の思想と同じなので許容でよい）。
- **Phase2 でランキング等のバックエンドを導入する際は必ず無効化/署名検証を設計に含める**
  （`_headers` のコメントにある connect-src 開放と同時に検討。この一行を architecture.md に明記）。

### 5-2. `style-src 'unsafe-inline'` の恒久化を避ける ★
Svelte の動的 inline style（進捗バー幅等）都合で必須なのは妥当。ただし将来
`nonce`/`'unsafe-hashes'` への移行余地があること、inline style 以外（`<style>` 注入）まで
許容されている点をリスクとして記録しておく。現実的な代替は CSS カスタムプロパティ経由
（`style:--w={pct}` → CSS 側で `width: var(--w)`）だが、Svelte の style ディレクティブも
inline style を生成するため効果は限定的。**現状維持で可、記録のみ**。

### 5-3. 依存関係の継続監査 ★★
外部通信ゼロでランタイム面は堅いが、ビルド時のサプライチェーン
（vite / svelte / workbox 等）が残る。GitHub の Dependabot alerts + `npm audit` を
CI に追加し、lockfile 更新を定期化する。

### 5-4. SW 更新方式（autoUpdate + skipWaiting）★
新 SW が即時制御を取るため、**プレイ中のセッションでアセット版がすれ違う**理論上の
リスクがある（実害は再読込で解消）。頻繁なデプロイを始めたら、
「更新があります・再読み込み」トースト方式（`registerType: 'prompt'`）への切替を検討。

### 5-5. 確認済みで問題なし（記録）
- `eval`/`Function` 不使用・`{@html}` 不使用・postMessage 不使用。
- localStorage 書込みは単一キー・JSON のみ。個人情報は一切保存しない。
- 共有（Web Share/クリップボード）はテキストのみで URL インジェクション面なし。
- `Permissions-Policy` でカメラ・マイク・位置情報・決済を明示拒否。

## 6. 免責・法務・プライバシー

### 6-1. プライバシーポリシーが無い ★★★
「データの保存について」の localStorage 説明はあるが、ポリシーとしては不足。
About（または独立画面）に以下を明記する:
- 個人情報・行動データを**収集・送信しない**こと（アナリティクス不使用）。
- Cookie 不使用・localStorage のみ（用途と消し方）。
- ホスティング（Cloudflare Pages）のアクセスログが技術的に発生しうる旨。
- 子ども向けサービスとして保護者向けの一文（広告なし・課金なし・外部送信なし）。

### 6-2. 免責事項が無い ★★★
学習コンテンツとして最低限の免責を About に追加する:
- 漢字の読み・意味・部品分解は辞書データ由来であり、**正確性・網羅性を保証しない**
  （教育利用時は教科書を優先する旨）。
- 本アプリの利用により生じた損害への免責。
- 外部リンク（ライセンス先等）の内容には責任を負わない旨。

### 6-3. アプリ本体のライセンス表記が曖昧 ★★
About に「アプリ本体のコードはこの限りではありません」とだけあり、実際のライセンス
（リポジトリの **MIT License, © 2026 fuji18**）が明記されていない。
「本アプリのソースコードは MIT License」＋著作権表示（© 2026 fuji18）＋
リポジトリへのリンク（公開するなら）を追記する。フッター等に `© 2026 fuji18` と
アプリバージョン（`package.json` の version を表示）もあると問い合わせ対応が楽になる。

### 6-4. 「ガチャ」表現について（確認結果: 問題なし・記録）★
課金・有償ガチャ要素が無いため景品表示法/資金決済法上の論点は無い。誤解防止のため
About に「本アプリに課金要素・広告はありません」と一文入れておくとより安心
（メタディスクリプションの「無料」と整合）。

### 6-5. クレジット文言の整合 ★
クレジットに「KANJIDIC2 **and KRADFILE**」の逐語があるが、直後で
「部品分解は KanjiVG を KRADFILE の**代替として**使用」とあり、KRADFILE を実際に
使っていないなら逐語クレジットから外す（または「雛形逐語のまま」の判断を
docs に残す）。EDRDG ライセンス的には過剰クレジットは無害だが、正確な方が良い。

### 6-6. お問い合わせ先 ★
不具合報告・データ出典に関する連絡先（GitHub Issues か連絡用メール）が無い。
CC BY-SA 継承物を配布している以上、指摘を受け付ける窓口を About に置くのが望ましい。

## 7. PWA・パフォーマンス

### 7-1. マニフェストアイコンの不足 ★★
アイコンが 512px `purpose: any` の 1 枚のみ（コード内コメントでも後続フェーズと明記）。
- `192x192` を追加（Lighthouse / 一部ランチャー要件）。
- `purpose: 'maskable'` 専用アイコン（セーフゾーン内にモチーフ）を追加。
- Windows/Chromebook 向けに `screenshots` があるとインストール UI が改善（任意）。

### 7-2. 辞書 JSON のロード最適化 ★
`kanji.json` 321KB + `combine-dict.json` 132KB + `parts.json` 69KB ≈ 計 520KB を
初回に fetch。Cloudflare 側の Brotli で実転送は小さく、プリキャッシュ済みのため
現状問題は無い。将来データを増やす場合はレベル別分割ロード（elementary を先行）を検討。

### 7-3. フォントサブセットの運用 ★
Yuji Syuku サブセットに含まれない字が UI 文言に増えるとフォールバック明朝と混在する。
サブセット生成をスクリプト化（対象文字列の抽出 → 再サブセット）して
`scripts/` に置き、文言変更に追随できるようにする。

## 8. 運用・その他

### 8-1. 未使用設定 `hintAlwaysOn` の整理 ★
`Settings.hintAlwaysOn` は型・migrate・既定値に存在するが、参照する UI/ロジックが無い。
実装予定が無いなら削除（migrate は「未知キー無視」で後方互換のまま）、
予定があるならチケット化して docs に紐付ける。デッドコードは仕様の誤読を招く。

### 8-2. ~~E2E の実行環境を CI に~~（訂正: 対応済みを確認）
`.github/workflows/ci.yml` が lint / typecheck / gen:data / verify:data / unit（カバレッジ閾値）/
build / Playwright E2E まで実行していることをチケット化時に確認した。**対応不要**。
残るのは依存監査（Dependabot / npm audit）のみ → 5-3 に統合。

### 8-3. デイリーお題のタイムゾーン仕様の明文化 ★
`todayYmdJst` で JST 固定は実装済みだが、海外からのアクセス時に「日本時間で切替わる」
ことを About か Home のツールチップで一言案内すると混乱がない（軽微）。
