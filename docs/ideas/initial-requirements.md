# 早起きは三文の得 (Hayaoki Sanmon) - アイデアメモ

## このドキュメントの位置づけ

このドキュメントはこれまでの壁打ち・ブレストの成果物をまとめたメモであり、MVP（Minimum Viable Product）開発のための設計図として機能します。

## プロダクトビジョン

「早起きは三文の得」をデジタルで体現し、不規則になりがちな現代人の生活リズムを整える。
シンプルな操作で朝の時間を充実させ、自己成長を実感できるプラットフォームを提供する。

**一言で言うと**: 朝活の習慣化と集中をサポートする、ゲーミフィケーション特化型タスク管理アプリ

---

## 解決したい課題

### 今困っていること

1. **二度寝してしまい、朝活が続かない**
* 朝の時間を使って資格勉強や家事をしたいが、モチベーションが起きずに寝てしまう
* 「やらなきゃ」というプレッシャーだけがあり、取り掛かるまでのハードルが高い


2. **作業に集中できない**
* せっかく早起きしても、スマホを触ってしまいダラダラと時間が過ぎてしまう
* 自分がどれくらい作業を進められたのか、時間的な実感がない


3. **頑張りが可視化されず、挫折しやすい**
* 数日早起きを頑張っても、記録が残らないため達成感が薄い
* 1日サボってしまった時の自己嫌悪で、そのまま習慣が途切れてしまう


4. **個人開発におけるサーバー維持費のプレッシャー**
* ユーザーが増えるほどBaaS（Firebase等）の利用料がかさみ、赤字になるリスクがある

## 主要機能の候補

### P0 (MVPで絶対必要)

#### 1. 毎朝のタスク設定機能

* 毎朝、ポモドーロを開始するタイミングでその日のタスクを設定（最大3つ）
* **最低1タスクから開始可能**。朝のハードルを下げるため、全て埋めなくても開始できる
* 「ジャンル（タグ）」＋「具体的なタスク内容（任意）」の組み合わせで登録
  * タグのみのタップで登録完了するため、テキスト入力を強制しない
  * 同じジャンル（例：学習）で複数のタスクを設定することも可能
* **前日のタスクをデフォルト表示**: 変更がなければそのまま開始でき、通常日は2タップで完結
* 変更したい日だけ個別に編集する

#### 2. 朝活の締め切りと1日の管理ルール

* ユーザーが「1日のリセット時間（例: 午前3:00）」と「朝活の目標時間（例: 午前8:00）」を任意で設定
* **救済措置**: 目標時間より**前**に開始した作業は、終了が目標時間を過ぎていても朝活実績として集計
* **制限**: 目標時間を過ぎてから開始した作業は実績として集計しない

#### 3. ポモドーロ時間計測機能

* タスクごとに25分（集中）＋5分（休憩）のタイマーを開始
* 一時停止機能を搭載
* **早期完了時**: 25分の途中でタスクが完了した場合、タイマーは終了させず、残りの時間で他の作業を行うよう画面上で促す

#### 4. シンプルなカレンダー表示機能

* 月間カレンダーの各マスに、達成した日は「達成マーク（スタンプ等）」を表示
* 達成した日の各マスに**起床タイム（タイマー初回開始時刻）**を表示
  * 「だんだん早起きになっている」変化が一目でわかる
  * 他のポモドーロアプリにはない、朝活アプリならではの記録
* 現在の連続達成日数（ストリーク）を表示
* ※詳細ポップアップ（タスク内訳・作業時間・獲得EXP）はP1で追加

#### 5. 朝活リマインダー通知

* **朝のリマインダー**: ユーザーが設定した「朝活の目標時間」の少し前にプッシュ通知を送信
  * 通知をタップするとアプリが起動し、そのままタスク設定へ遷移
* **前夜リマインダー**: 毎晩（例: 22:00）に「明日のタスクを決めておきましょう」と通知
  * 前日夜にタスクを設定 → 翌朝は2タップで即開始できるフローを促進
  * 「やること決めてある」安心感が二度寝防止にも寄与
* 両通知ともユーザーが時刻・ON/OFFを設定可能

#### 6. レベルアップとストリーク機能

##### EXP設計

* **基本EXP**: 1ポモドーロ完了で「25 EXP」を獲得
* **ストリーク判定**: デイリー継続（その日の朝活を達成すれば+1日）
* **段階的ストリーク倍率**:

| ストリーク日数 | EXP倍率 | 1ポモドーロあたり |
|--------------|--------|----------------|
| 0〜2日 | × 1.0 | 25 EXP |
| 3〜6日 | × 1.5 | 37 EXP |
| 7〜13日 | × 2.0 | 50 EXP |
| 14日以上 | × 2.5 | 62 EXP |

* **朝限定EXP**: EXPは「目標時間前に開始した朝活」にのみ付与
  * 目標時間を過ぎてから開始した作業は記録できるが、EXPは獲得できない
  * 「朝じゃないと意味がない」をゲームメカニクスで明示し、朝活動機を強化
* **ペナルティ**: 連続達成が途切れてもレベルダウンはせず、倍率が× 1.0に戻るのみ

##### 再起ボーナス

* ストリークが途切れた翌日に朝活を達成すると **+100 EXP のボーナス** を獲得
* 「サボった自己嫌悪」を「取り返せる前向きな気持ち」に変換する設計

##### レベル曲線

序盤に達成感を得やすくし、長期的なモチベーション維持を両立させる。

| レベル帯 | 必要EXP/レベル | 累計EXP |
|---------|-------------|--------|
| Lv1〜5 | 150 EXP | 750 EXP |
| Lv6〜10 | 300 EXP | 2,250 EXP |
| Lv11〜15 | 500 EXP | 4,750 EXP |
| Lv16〜20 | 650 EXP | 8,000 EXP |

* Lv5到達目安: 開始後約1週間（序盤の達成感を早期に体験させる）
* Lv20到達目安: アクティブユーザー（週4日・3ポモドーロ/日）で約3ヶ月
  * 成功指標「Lv20到達率30%以上（3ヶ月）」と整合

##### 称号システム

レベル帯ごとに称号が変わり、数値以上の達成感を演出する。

| レベル | 称号 |
|-------|------|
| Lv1〜5 | 朝活見習い |
| Lv6〜10 | 朝活の習慣者 |
| Lv11〜15 | 早起きの達人 |
| Lv16〜20 | 早起き王 |
| Lv21〜 | 早起きは三文の得 |

* ※継続動機の核心となる機能のため、カレンダーと同時にMVPに含める

### P1 (重要だけど後回しでもいい)

#### 7. カレンダー詳細ポップアップ

* スタンプをタップすると詳細（タスク内訳、各作業時間、合計作業時間、獲得経験値）がポップアップ表示

#### 8. データバックアップ機能（Google Drive手動連携）

* Storage Access Framework (SAF) を用いた、ユーザー自身のGoogle Driveへの手動エクスポート/インポート機能
* ※Auto Backup（Android標準）はP0リリース時点で有効化しておく

### P2 (あったら嬉しい)

#### 9. 収益化・マネタイズ機能

* **投げ銭機能**: 開発者を応援する少額課金ボタン（基本無料を維持）
  * **表示タイミング**: レベルアップ演出の直後に「開発者を応援する ☕」ボタンをオプション表示
  * ユーザーが最も満足している瞬間に出すことで、心理的ハードルを下げる

* **リワード動画広告**: 視聴することで「ストリークが途切れるのを1日防ぐ（ストリークフリーズ）」救済措置
  * **フリーズ使用時のEXP倍率**: ストリーク日数・倍率ともに維持する
  * 倍率を維持することでフリーズの価値を高め、リワード広告の視聴動機を最大化する

* **バナー広告**: カレンダー画面下部、設定画面下部のみに配置（**タスク設定・タイマー画面は広告排除**）
  * カレンダー詳細ポップアップ表示中はバナー広告を非表示にする

## UXフロー設計

### 朝のメインフロー（通常日）

```
朝活リマインダー通知
  ↓ タップ1
【タスク設定画面】
  前日のタスクがデフォルト表示（ジャンルタグ + テキスト）
  変更不要なら「開始」ボタンをタップ
  ↓ タップ2
【タイマー画面】
  1つ目のタスクのポモドーロ開始
```

**通常日（タスク変更なし）**: 通知タップ → 開始 = **2タップ**

---

### 各シーンのタップ数

| シーン | 操作 | タップ数 |
|--------|------|--------|
| 通常日（前日タスクをそのまま使う） | 通知 → 開始 | 2タップ |
| 一部変更したい日 | 通知 → 該当タスク編集 → 開始 | 3〜4タップ |
| 初日・全て変更したい日 | タグ3つタップ → 開始 | 4タップ |
| 急いでいる日（タスク1つだけ） | 通知 → タグ1つタップ → 開始 | 3タップ |

---

### タスク設定のルール

* **最低タスク数**: 1つから開始可能（3つ全て埋めなくてもよい）
* **デフォルト値**: 前日設定したタスクを引き継ぐ
* **タスク形式**: ジャンルタグ（必須） + テキスト（任意）
  * タグのみでも登録完了 → テキスト入力を強制しない
* **最大タスク数**: 3つまで

| 毎朝タスクを決めるフロー | ✅ | ❌（事前登録が主） | △ | ❌ |
| 朝限定EXP（朝以外は記録のみ） | ✅ | ❌ | ❌ | ❌ |
| 起床タイムの記録・可視化 | ✅ | ❌ | ❌ | ❌ |
| 挫折させないEXP設計 | ✅ | ❌ | ❌ | △ |

| 運営ランニングコスト | 0円(✅) | - | - | - |

### このツールの強み

1. **「朝」に特化した独自ルール**
* 締め切り時間による適度なプレッシャーと、ギリギリでも始めればOKという救済措置の絶妙なバランス。
* 朝限定EXPにより「朝にやらないと損」というゲームメカニクスで朝活動機を強化。

2. **起床タイムの可視化**
* カレンダーに毎日の起床タイムを記録し、「だんだん早起きになっている」変化を実感できる。他のポモドーロアプリにはない、朝活アプリならではの体験。

3. **前夜から始まる朝活サポート**
* 前夜リマインダーでタスクを事前設定 → 翌朝は2タップで即開始。「朝起きてから考える」手間をなくし、二度寝を防ぐ。

4. **挫折を防ぐポジティブな設計**
* サボってもレベルが下がらない。再起ボーナスで「取り返せる」前向きな気持ちに変換。段階的EXP倍率で少しの継続でも報われる。

5. **集中を絶対に邪魔しない広告設計**
* 収益化を図りつつも、タイマー実行中など一番大事な画面には一切広告を出さない徹底したUXへの配慮。

6. **完全ランニングコストゼロ**
* サーバーレス、BaaSレス、Google Drive引き継ぎにより、個人開発者が赤字リスクゼロで永続的に運営可能。

## 技術的な検討事項

### データの保存方法

**結論**: SQLite（Android標準のRoomライブラリ）

* オフラインで完結し、レスポンスが高速。毎月のサーバー代が一切かからない。
* カレンダー表示のための日別データの集計・検索が容易。

**テーブル設計（方針）**:

```
sessions テーブル（1ポモドーロ = 1レコード）
  - id
  - date          // 朝活の日付
  - task_id
  - started_at    // 開始時刻
  - completed_at  // 完了時刻
  - is_morning    // 朝活時間内かどうか（EXP付与の判定）
  - exp_earned

daily_records テーブル（1日 = 1レコード）
  - date
  - wake_time     // その日最初のsessionのstarted_at（カレンダーの起床タイム表示用）
  - total_exp
  - streak_count
  - is_achieved   // 朝活達成フラグ
```

### タイマーのバックグラウンド動作

* **Foreground Service**: OSのバッテリー最適化（Dozeモード等）によってタイマーが強制終了されるのを防ぐため、通知欄にタイマーの残り時間を常駐・更新させる。
* **一時停止中の扱い**: 一時停止中もServiceを継続し、通知に「一時停止中」と明示する（バッテリー消費とUXのバランスを取る）
* **中間保存と復元**: 計測中のデータを5分ごとにDBへ保存し、アプリが強制終了された場合は再起動時に「前回の計測を復元しますか？」ダイアログを表示
* **バッテリー最適化除外の誘導**: 初回起動時にユーザーへ「バッテリー最適化の除外設定」を促すダイアログを表示（Xiaomi・Samsung等のメーカー独自のタスクキル対策）

### バックアップ・データ連携

* Google Cloud等のAPIは使わず、Android標準の `Auto Backup` と `Storage Access Framework (SAF)` を用いて実装コストと維持費を極小化する。

### 広告実装

* Google AdMob SDKを導入し、バナー広告とリワード広告を実装する。

## 非機能要件（とりあえず）

### パフォーマンス

* アプリ起動時間: 2秒以内（朝の眠い時間帯にストレスを与えない）
* カレンダーの月切り替え: 0.5秒以内

### 使いやすさ

* 初めて使う人でも直感的に操作可能なUI（アプリを開いてから3タップ以内でタイマーを開始できる）
* ライトモード/ダークモードへの対応（朝の暗い部屋、明るい部屋両方での視認性確保）

### 信頼性

* 完全オフライン動作（ネットワーク切断時でもタイマーや記録が正常に行われる）
* データの不整合が起きない（Roomのトランザクション活用）
